# 产品定义思考：Agents Harness Engineering System

> 日期：2026-05-19  
> 状态：产品定义约束版 v0.2
> 说明：本文由原始语音思考整理而来，并在后续市场、竞品、架构分析基础上进一步修订。修订目标不是追加新观点，而是把更顶层的产品判断内化为可验证、可验收、可推进的约束。部分语音识别不确定的专有名词仍保留为“待核对”，避免过度修正。

## 1. 核心问题

我们要定义的不是一个简单的“AI 聊天工具”，也不是把 Codex、Claude Code、OpenHands 这类工具原样暴露给用户。

真正的问题是：

**在合法合规、可商用、可持续演进的前提下，我们能不能做出一套面向下游客户的 Agents Harness Engineering System，把编码、调试、运维、代码理解等工程能力，从专家工具转化为普通 AI 用户、半专业用户和专业用户都能理解、执行、验收的工程任务系统。**

这个产品需要回答三个问题：

1. 我们服务谁？
2. 我们要向用户暴露多少能力？
3. 我们选择什么样的产品形态和技术底座，既能交付第一期，又能承载后续成长？

这里的核心分歧，不应被简化为“选择 DeepSeek-TUI 还是 OpenHands”。更准确地说，DeepSeek-TUI 代表的是**最快可交付的技术形态**，OpenHands 代表的是**更接近产品化工作区的底座形态**。二者都可能有价值，但它们回答的是不同层级的问题：

- DeepSeek-TUI 能否快速验证模型、CLI、沙箱和工程执行能力。
- OpenHands 能否承载半专业用户所需的 Web 工作区、任务状态、执行过程、变更展示和验收闭环。
- Agents Harness Engineering System 最终要回答的是：哪种形态能让目标客户真正用起来、信任它、复用它，并愿意为它付费。

因此，后续推进不能停留在“哪个技术更强”的讨论，而要把分歧转化为可验证实验：同一批真实任务、同一批目标用户、同一组验收指标，比较不同底座和交互形态的实际表现。

## 2. 产品定义者的责任

作为产品开发者和产品定义者，我们不能只从“技术是否强大”出发做选型，而要对用户负责。

责任边界取决于用户画像：

- 如果用户已经是精通型开发者，可以直接使用 Codex、Claude Code、Cursor、科大讯飞/其他闭源或商业编码工具（待核对）等高阶工具。
- 如果用户只是停留在 Chat AI 阶段，直接给他一个复杂的 coding agent 或工程系统，可能并不能形成有效生产力。
- 如果用户是半专业或专业但不精通的群体，我们需要把复杂能力包装成统一入口、统一交互、统一验收的产品。

因此，产品定义的责任不是“把所有能力都暴露出来”，而是：

**在能力、复杂度、可控性、商业可用性之间找到平衡，并把工程工作流转化为用户能理解、能执行、能验收的产品体验。**

这份责任还包括对自身判断方式的约束：市场调研、LLM 分析、竞品拆解和架构研究都只是帮助我们压缩不确定性的手段，不能替代真实任务和真实用户的验证。产品定义过程可以借助 Claude、GPT、Codex、Claude Code 等工具放大个人的分析、设计和架构能力，但最终判断必须落到可观察行为上。

如果后续继续产出报告，却没有形成任务原型、用户实验和验收数据，那么高质量分析会变成新的风险。每一次分析都必须回答一个推进问题：

- 它是否缩小了用户范围？
- 它是否澄清了任务边界？
- 它是否降低了技术选型风险？
- 它是否产出了可以验证的指标？
- 它是否改变下一步要做的实验？

如果答案都是否定的，就不应继续扩大分析范围。

## 3. 当前市场和技术判断

### 3.1 开源与闭源工具的差异

OpenHands 这类开源方案有价值，可以作为原型机或产品形态起点。它的意义在于：

- 提供可观察、可改造的开源架构。
- 在开源与闭源之间探索商业可用性。
- 让我们可以基于现有框架做定制化适配、迁移和产品化改造。

但它和 Codex、Claude Code 等更成熟的闭源/商业 agent coding 产品之间，仍然存在明显差距：

- 产品体验差距。
- 工程能力差距。
- 商业交付成熟度差距。
- 对复杂工程任务的稳定性差距。

所以不能因为选择 OpenHands 作为起点，就放弃其他技术路线。OpenHands 更适合被视为“站在巨人肩膀上的第一步”，不是最终答案。

同样，也不能因为 DeepSeek-TUI 更轻、更直接、更容易接入国内模型，就把它直接等同于产品答案。它可以是很好的技术验证器、CLI 原型和专业用户入口，但不一定适合承担半专业用户的第一交互界面。

当前阶段应把候选底座分成三类，而不是混成一个选择题：

- **技术验证器**：例如 DeepSeek-TUI，用于验证模型接入、命令执行、沙箱、安全边界和 CLI 任务完成能力。
- **产品工作区底座**：例如 OpenHands，用于验证 Web 交互、会话、任务状态、执行过程、代码变更、终端和浏览器预览是否能形成可用闭环。
- **能力标杆**：Codex、Claude Code、Cursor 等闭源或商业产品，用于校准体验、能力上限和商业成熟度，不直接作为短期可控底座。

### 3.2 我们的优势

我们的优势不一定在前端交互工具本身，而在后端资源和可控性：

- 自研算力。
- 自主集群。
- token 成本和供给可覆盖。
- 可以接入国内模型体系，如 DeepSeek、MiniMax、GLM、Qwen 等。
- 甚至可以定制自己的模型后端服务体系。

这意味着我们真正需要的是：

**一个 80-90 分的 Codex / Claude Code / Cursor 形态的产品前端，把自有算力、模型和工程工作流包装成可交付服务。**

这里的“80-90 分”不能只按界面相似度衡量，而要按目标用户完成任务的能力衡量。一个合格的第一期产品前端至少要做到：

- 用户知道自己该输入什么。
- 系统知道什么时候需要澄清边界。
- 用户能看懂系统准备怎么做。
- 用户能看到执行过程中的关键状态，而不是被日志淹没。
- 用户能知道结果如何验收。
- 失败时系统能给出下一条可执行路径。

## 4. 初步用户画像

当前阶段假设的用户，不是精通型工程师，而是：

- 基本停留在 Chat AI 使用阶段的用户。
- 习惯用豆包、通义、DeepSeek、Kimi 等聊天式产品提问。
- 通过文字、截图、日志片段获得建议。
- 有简单业务系统开发、轻量运维、bug 调试、代码阅读、代码理解需求。
- 具备一定业务判断，但不一定懂完整软件工程流程。
- 不想面对大量代码、报错栈、debug log 和复杂工具链。

这类用户真正需要的不是“更多技术细节”，而是：

- 一个统一入口。
- 一个能理解目标的系统。
- 一个能拆任务、拆步骤、定边界的系统。
- 一个能告诉他怎么验收结果的系统。
- 一个在失败时能给出下一步探索路径的系统。

这类用户的关键特征不是“完全不会技术”，而是**缺少稳定的软件工程工作流**。他们可能能描述业务目标、识别结果是否符合业务预期，也可能能提供截图、日志、仓库或运行环境，但他们不一定知道如何把这些信息组织成一次完整的工程任务。

因此，产品不能只问“用户会不会写代码”，还要问：

- 用户能否清楚描述目标？
- 用户能否提供必要上下文？
- 用户是否愿意授权系统读取仓库、日志或运行环境？
- 用户是否能理解系统给出的验收标准？
- 用户是否能接受系统在不确定时暂停并要求补充信息？

这些问题后续必须通过真实任务验证，而不是在文档里假设成立。

## 5. 产品定位

产品应定位为：

**面向半专业和专业用户的 Agents Harness Engineering System，用统一入口承载 coding、debug、运维、代码理解和轻量工程任务，并通过计划、执行、测试、验收、迭代闭环降低用户使用 AI 工程能力的门槛。**

它不应该只是一个聊天窗口，也不应该只是一个 IDE 插件。

更准确地说，它是一套“工程任务约束与执行系统”：

- 用户表达目标。
- 系统澄清边界。
- 系统拆解任务。
- 系统生成待办。
- 系统执行或指导执行。
- 系统提出测试与验收标准。
- 系统判断是否达到终止条件。
- 系统在失败时给出下一轮修复路径。

产品的第一阶段不应宣称自己是通用 AgentsOS，也不应把“能调用工具”和“能完成工程任务”混为一谈。更稳妥的第一定位是：

**面向半专业用户的工程任务诊断与修复闭环系统。**

它的第一价值不是“AI 会写代码”，而是把一次工程问题变成可管理的任务链：

`输入问题 → 澄清上下文 → 形成计划 → 执行或指导执行 → 生成变更 → 运行验证 → 输出验收标准 → 给出失败分支`

只有这条链路被验证之后，再讨论更大的 AgentsOS、Skill 生态、多入口、多模型调度和企业控制平面，才是有根基的。

## 6. 关键产品原则

### 6.1 不追求无限发散，优先约束边界

这个系统不应该追求“什么都能聊、什么都能答”。对目标用户来说，更重要的是：

- 能不能把问题限定在可解决范围内。
- 能不能清楚说明当前上下文不足在哪里。
- 能不能避免基于片面信息给出过度自信的答案。
- 能不能把复杂任务拆成可验证步骤。

边界约束本身要成为产品能力，而不是隐藏在 prompt 里的风格偏好。系统应该在以下场景主动收敛：

- 上下文不足时，不直接给出确定性结论。
- 任务范围过大时，先拆出最小可验证子任务。
- 用户提出模糊目标时，先把目标改写为可验收结果。
- 涉及生产环境或高风险操作时，默认进入人工确认。
- 无法验证结果时，明确说明“未验收”，而不是声称完成。

### 6.2 信息密度要高，但只提供正确的信息

普通用户不需要看到所有代码、日志和底层状态。

产品要做的是提升“有效信息密度”：

- 只展示用户当前需要决策的信息。
- 隐藏不必要的底层噪音。
- 对复杂信息做结构化摘要。
- 把技术输出转化成“目标、步骤、风险、验收、下一步”。

信息密度的衡量标准不是页面上展示的信息越少越好，而是用户能否在不被底层噪音淹没的情况下做出下一步判断。第一期应把信息分成三层：

- **决策层**：目标、当前状态、下一步、风险、是否需要用户确认。
- **验收层**：测试结果、运行现象、变更摘要、成功/失败判定。
- **证据层**：命令输出、日志片段、diff、文件路径、错误堆栈，默认折叠但可展开。

如果用户必须阅读大段日志才能知道任务是否完成，说明产品没有完成它该承担的翻译工作。

### 6.3 验收链条必须闭合

传统 Chat 模式最大的问题是：LLM 会告诉用户“这样修就好了”，但它未必接入运行环境、生产环境、完整日志和真实上下文。

因此，系统不能只输出修复建议，还必须输出：

- 修复后应该看到什么现象。
- 看到什么现象才说明修好了。
- 如果没有出现预期现象，下一步检查什么。
- 如果出现另一类现象，可能对应哪种替代方案。
- 当前判断的依据和不确定性在哪里。

产品的理想输出不是单一路径答案，而是更可靠的探索结构：

1. 最大概率修复方案。
2. 对应验收标准。
3. 失败时的第二路径。
4. 失败时的第三路径。
5. 每一路径的观察指标和终止条件。

验收链条必须可记录、可回放、可评分。每次任务结束时，系统至少要产出：

- 本次任务目标。
- 已执行步骤。
- 已修改内容或已生成建议。
- 已运行的验证动作。
- 验证结果。
- 未验证事项。
- 用户下一步需要确认什么。
- 如果失败，下一条探索路径是什么。

如果没有这些输出，即使模型生成了正确代码，也不能算完整交付。

### 6.4 软件工程工作流要产品化

系统需要把成熟的软件工程工作流内化进产品：

- 目标澄清。
- 范围约束。
- 任务拆解。
- todo list。
- TDD / PDD / 测试优先思想（原文缩写待核对）。
- 执行过程跟踪。
- QA 准出。
- 验收标准。
- 迭代终止条件。

这一部分可以吸收 Superpowers 的工作流思想。对大多数目标用户来说，具备完整的软件工程思维工作流，已经足够形成明显价值。

但工作流产品化不是把所有高阶能力一次性暴露出来。第一期应优先吸收最基础、最能降低失败率的工程纪律：

- 先澄清目标，再执行。
- 先定义验收，再声称完成。
- 任务拆成可检查 todo。
- 每一步有状态，而不是黑箱运行。
- 失败时回到证据，而不是换一种说法继续猜。

复杂 Skill 市场、深度自动化编排、跨组织知识沉淀等能力可以作为后续扩展，不应抢占 MVP 的复杂度预算。

### 6.5 分析必须转化为实验

产品定义阶段的分析价值，最终要体现在实验设计上。后续每一个重要判断，都应转化为以下格式：

`假设 → 原型 → 任务样本 → 目标用户 → 验收指标 → 继续/调整/停止标准`

例如：

- 假设：半专业用户不适合直接使用 TUI。
- 原型：DeepSeek-TUI 轻包装版 vs OpenHands Web 工作区版。
- 任务样本：日志诊断、小 bug 修复、代码理解、轻量运维。
- 目标用户：内部实施/运维/技术支持/半专业研发人员。
- 验收指标：首次成功任务时间、人工解释负担、验收闭环率、复用意愿。
- 决策：谁在真实任务里更快、更稳、更少依赖旁人解释，谁就是第一期更合适的形态。

## 7. 产品形态选项

产品形态需要分阶段思考。

第一期不一定要覆盖所有入口，但选型时必须判断底座能否承载未来扩展。

可能形态包括：

- Web 端：长对话、任务管理、可视化流程、验收结果展示。
- IDE 插件：例如 VS Code 插件，用于更专业用户的代码上下文接入。
- CLI：适合开发者或内部团队使用。
- 桌面 GUI：降低安装和本地环境使用门槛。
- IM 接入：飞书、企业微信等，用于企业内部协作和轻量任务入口。
- 工具调用体系：接入代码仓库、运行环境、日志系统、工单系统、CI/CD 等。

关键不是一期做哪个入口，而是：

**我们选择的 harness 底座有没有能力承载这些形态的延展。**

如果底座只能交付一次，后续无法演进，那么产品很快会被限制住。

因此，第一期产品形态不应只看“能不能跑起来”，而要看它是否具备长大的空间。产品形态的评估应至少包含：

- 是否能支持任务状态展示。
- 是否能支持执行过程证据回放。
- 是否能支持用户确认和中断。
- 是否能支持验收结果结构化。
- 是否能支持未来接入 IDE、CLI、IM 或 API。
- 是否能支持企业部署下的权限、审计和数据隔离。

在这个标准下，TUI/CLI 更适合承担专业用户和技术验证入口，Web 工作区更适合承担半专业用户的主入口。二者不是非此即彼，但第一期必须明确主入口，否则体验会被撕裂。

## 8. 技术选型关注点

### 8.1 LLM 后端

后端 LLM 需要以主力模型为基准，再配合其他国内开源或商业模型进行能力补充。

当前可考虑的方向包括：

- DeepSeek 作为主力模型之一（原文“deep sick”应为 DeepSeek）。
- MiniMax、GLM、Qwen 等国内模型作为补充（具体版本待核对）。
- 自有模型后端服务体系。
- 后续再考虑模型路由、调度、编排和多模型协同。

第一阶段可以暂不把复杂调度作为核心问题，而是先聚焦上层 Agents Harness System 的产品闭环。

模型选型的第一验收标准不是 benchmark，而是能否稳定完成目标任务样本。第一阶段应建立小规模内部任务集，用同一套任务比较不同模型组合：

- 日志诊断是否能找出高概率根因。
- 小型 bugfix 是否能生成可运行修改。
- 代码理解是否能给出准确文件路径和调用链。
- 轻量运维任务是否能给出安全可执行步骤。
- 失败时是否能承认不确定，并提出下一步证据需求。

只有在任务闭环稳定后，才值得引入复杂模型路由和多模型协同。

### 8.2 Harness 底座

底座需要评估：

- 是否支持任务拆解。
- 是否支持工具调用。
- 是否支持上下文管理。
- 是否支持长任务执行。
- 是否支持执行过程可观测。
- 是否支持测试和验收机制。
- 是否支持多入口扩展。
- 是否支持企业级部署和合规要求。
- 是否便于二次开发和定制化。

OpenHands 可作为起点，但需要和 Codex、Claude Code、Cursor、以及其他闭源方案进行能力对照。

这里需要避免两个误区：

- 不应因为 OpenHands 更产品化，就默认它是最优底座。
- 不应因为 DeepSeek-TUI 更轻、更容易接国产模型，就默认它足以成为客户产品。

更合理的做法是把底座选择变成双原型验证：

| 原型 | 定位 | 验证重点 | 不承担的责任 |
|---|---|---|---|
| DeepSeek-TUI 轻包装版 | 技术验证器 / CLI 原型 | 模型接入、命令执行、沙箱、专业用户效率 | 不证明半专业用户主入口成立 |
| OpenHands Web 工作区版 | 产品工作区原型 | 任务流、状态展示、变更查看、终端/浏览器证据、验收闭环 | 不证明底层能力已达闭源标杆 |

两条原型都不应被包装成最终答案。它们的任务是帮助我们用真实数据判断第一期产品形态。

## 9. MVP 建议

第一期不应追求完整平台，而应验证最核心的闭环：

**用户输入一个工程目标，系统能把它转化为可执行、可跟踪、可验收的工程任务。**

更具体地说，MVP-01 应聚焦：

**面向半专业用户的工程任务诊断与修复闭环。**

第一批任务不应泛化为“什么都能做”，而应控制在四类：

1. 日志/报错诊断：用户提供错误日志、截图或终端输出，系统给出根因假设、修复路径和验收方法。
2. 小型 bug 修复：用户提供仓库或片段，系统定位相关文件，生成补丁或 PR，并运行最小验证。
3. 代码理解：用户提出“这个功能在哪里、为什么这样运行、怎么改”的问题，系统给出文件、调用链和风险点。
4. 轻量运维任务：用户提供环境信息，系统生成安全步骤、确认点和回滚建议。

建议 MVP 包含：

1. 统一 Web 入口。
2. 目标输入与澄清。
3. 任务拆解。
4. todo list 生成。
5. 测试/验收标准生成。
6. 执行过程状态展示。
7. 结果验收与失败分支建议。
8. 基础工具调用能力。

先不做或弱化：

- 复杂多模型路由。
- 完整 IDE 插件生态。
- 飞书/企业微信入口。
- 高度自动化生产环境接入。
- 过重的 skill 体系复杂度。

### 9.1 MVP 验收指标

MVP 是否成立，不能靠“看起来像 Codex / Claude Code / Cursor”判断，而要靠真实任务数据判断。

建议第一轮 dogfood 使用 20-30 个真实任务样本，并记录以下指标：

| 指标 | 定义 | 第一轮可接受门槛 |
|---|---|---|
| 首次成功任务时间 | 用户从输入目标到看到可验收结果的时间 | 简单任务 15 分钟内，中等任务 45 分钟内 |
| 任务闭环率 | 系统完成“计划、执行、验证、验收说明”的任务比例 | ≥ 60% |
| 验收可理解率 | 用户不需要工程师解释，也能判断结果是否完成的比例 | ≥ 70% |
| 人工解释负担 | 旁观工程师需要额外解释产品状态或下一步的次数 | 每个任务 ≤ 2 次 |
| 失败可恢复率 | 任务失败后，系统能给出有效下一步证据需求或替代路径的比例 | ≥ 70% |
| 有效上下文请求率 | 系统要求补充的信息确实影响后续判断的比例 | ≥ 70% |
| 用户复用意愿 | 用户完成任务后愿意把下一次类似任务继续交给系统 | ≥ 50% |
| 成本可控性 | 单任务 token / 推理 / 沙箱成本是否可解释、可接受 | 每类任务形成成本区间 |

这些门槛不是最终商业标准，而是第一期是否值得继续投入的最低证据。

### 9.2 双原型对照实验

围绕 DeepSeek-TUI 与 OpenHands 的分歧，第一轮不应靠讨论决策，而应靠对照实验决策。

实验设计：

- 用同一批 20-30 个真实任务样本。
- 让 DeepSeek-TUI 轻包装版和 OpenHands Web 工作区版分别完成。
- 记录相同指标。
- 观察用户是否能独立完成输入、理解过程、验收结果和处理失败。

决策规则：

- 如果 DeepSeek-TUI 在任务完成率、速度和解释负担上明显领先，可将其作为第一期专业用户入口，并另行设计半专业用户包装。
- 如果 OpenHands Web 工作区在验收理解、状态可见性和复用意愿上明显领先，可将其作为第一期主入口。
- 如果两者都表现不佳，说明问题不是底座选择，而是用户场景、任务边界或交互设计还不够收敛。

## 10. 需要继续确认的问题

### 用户与场景

- 第一批用户到底是内部团队、企业客户，还是外部 SaaS 用户？
- 他们最常见的前三类任务是什么？
- 他们是否愿意提供代码仓库、日志、运行环境等上下文？
- 他们对自动执行的信任边界在哪里？

### 产品形态

- 第一期主入口是 Web、IDE 插件，还是二者并行？
- 用户是否需要“完全看不见代码”的体验？
- 哪些信息必须展示，哪些信息必须隐藏？
- 任务状态、验收标准、失败路径如何可视化？

### 技术底座

- OpenHands 能否满足二次开发、企业部署和多入口扩展？
- 是否需要同时验证 Codex / Claude Code / Cursor 类产品能力作为对照？
- 工具调用体系第一期接哪些工具最有价值？
- 是否要先做一个内部 dogfood 版本？

### 商业交付

- 我们卖的是工具、平台，还是带服务的工程能力？
- 交付边界在哪里？
- 如何定义“完成一次工程任务”的可计费或可衡量指标？
- 合规和数据隔离要求如何影响部署形态？

### 实验与准出

- 第一批 20-30 个真实任务从哪里来？
- 谁代表目标用户参与测试？
- 是否允许系统访问真实仓库、日志和运行环境？
- 哪些任务必须人工确认才能执行？
- 哪些指标达不到时必须停止继续扩大范围？

## 11. 当前阶段结论

这份思考的核心不是“选择哪个开源项目”，而是要确认一个更大的产品判断：

**我们要把 coding agent 能力从专家工具，转化为可被普通/半专业用户使用的工程任务系统。**

产品的关键竞争力不只来自模型能力，而来自：

- 自有算力与模型后端。
- 对目标用户复杂度的控制。
- 对工程工作流的产品化。
- 对测试、验收、准出的闭环设计。
- 对未来多入口、多工具、多模型扩展的承载能力。

同时，这份思考也要约束后续动作：不要继续无限扩展报告，不要把技术分歧伪装成产品结论，不要在没有真实任务数据的情况下宣布底座选择。

下一步最重要的工作，是把这套思考压缩成一个可执行的 MVP-01 验证计划：

1. 一页产品定位。
2. 三个核心用户场景。
3. 一条 MVP 工作流。
4. 一张底座选型评估表。
5. 一组 20-30 个真实任务样本。
6. 一套 DeepSeek-TUI 与 OpenHands 的双原型对照指标。
7. 一个明确的继续、调整或停止标准。

如果这些验证指标不能被建立起来，说明当前阶段还不应进入大规模产品开发；如果这些指标在真实任务中跑通，Agents Harness Engineering System 才真正从“好的产品想法”进入“可以继续投入的产品机会”。
