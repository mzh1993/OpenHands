# 产品定义思考：Agents Harness Engineering System

> 日期：2026-05-19  
> 状态：语音思考整理版 v0.1  
> 说明：本文先把原始口述稿整理为可继续讨论的产品定义稿。部分语音识别不确定的专有名词保留为“待核对”，避免过度修正。

## 1. 核心问题

我们要定义的不是一个简单的“AI 聊天工具”，也不是把 Codex、Claude Code、OpenHands 这类工具原样暴露给用户。

真正的问题是：

**在合法合规、可商用、可持续演进的前提下，我们能不能做出一套面向下游客户的 Agents Harness Engineering System，让普通 AI 用户、半专业用户和专业用户都能在受控边界内完成编码、调试、运维、代码理解等任务。**

这个产品需要回答三个问题：

1. 我们服务谁？
2. 我们要向用户暴露多少能力？
3. 我们选择什么样的产品形态和技术底座，既能交付第一期，又能承载后续成长？

## 2. 产品定义者的责任

作为产品开发者和产品定义者，我们不能只从“技术是否强大”出发做选型，而要对用户负责。

责任边界取决于用户画像：

- 如果用户已经是精通型开发者，可以直接使用 Codex、Claude Code、Cursor、科大讯飞/其他闭源或商业编码工具（待核对）等高阶工具。
- 如果用户只是停留在 Chat AI 阶段，直接给他一个复杂的 coding agent 或工程系统，可能并不能形成有效生产力。
- 如果用户是半专业或专业但不精通的群体，我们需要把复杂能力包装成统一入口、统一交互、统一验收的产品。

因此，产品定义的责任不是“把所有能力都暴露出来”，而是：

**在能力、复杂度、可控性、商业可用性之间找到平衡，并把工程工作流转化为用户能理解、能执行、能验收的产品体验。**

## 3. 当前市场和技术判断

### 3.1 开源与闭源工具的差异

OpenHands 这类开源方案有价值，可以作为原型机或产品形态起点。它的意义在于：

- 提供可观察、可改造的开源架构。
- 在开源与闭源之间探索商业可用性。
- 让我们可以基于现有框架做定制化适配、迁移和产品化改造。

但它和 Codex、Claude Code 等更成熟的闭源/商业 agent coding 产品之间，仍然存在明显差距：

- 产品体验差距。
- 工程能力差距。
- 商业交付成熟度差距。
- 对复杂工程任务的稳定性差距。

所以不能因为选择 OpenHands 作为起点，就放弃其他技术路线。OpenHands 更适合被视为“站在巨人肩膀上的第一步”，不是最终答案。

### 3.2 我们的优势

我们的优势不一定在前端交互工具本身，而在后端资源和可控性：

- 自研算力。
- 自主集群。
- token 成本和供给可覆盖。
- 可以接入国内模型体系，如 DeepSeek、MiniMax、GLM、Qwen 等。
- 甚至可以定制自己的模型后端服务体系。

这意味着我们真正需要的是：

**一个 80-90 分的 Codex / Claude Code / Cursor 形态的产品前端，把自有算力、模型和工程工作流包装成可交付服务。**

## 4. 初步用户画像

当前阶段假设的用户，不是精通型工程师，而是：

- 基本停留在 Chat AI 使用阶段的用户。
- 习惯用豆包、通义、DeepSeek、Kimi 等聊天式产品提问。
- 通过文字、截图、日志片段获得建议。
- 有简单业务系统开发、轻量运维、bug 调试、代码阅读、代码理解需求。
- 具备一定业务判断，但不一定懂完整软件工程流程。
- 不想面对大量代码、报错栈、debug log 和复杂工具链。

这类用户真正需要的不是“更多技术细节”，而是：

- 一个统一入口。
- 一个能理解目标的系统。
- 一个能拆任务、拆步骤、定边界的系统。
- 一个能告诉他怎么验收结果的系统。
- 一个在失败时能给出下一步探索路径的系统。

## 5. 产品定位

产品应定位为：

**面向半专业和专业用户的 Agents Harness Engineering System，用统一入口承载 coding、debug、运维、代码理解和轻量工程任务，并通过计划、执行、测试、验收、迭代闭环降低用户使用 AI 工程能力的门槛。**

它不应该只是一个聊天窗口，也不应该只是一个 IDE 插件。

更准确地说，它是一套“工程任务约束与执行系统”：

- 用户表达目标。
- 系统澄清边界。
- 系统拆解任务。
- 系统生成待办。
- 系统执行或指导执行。
- 系统提出测试与验收标准。
- 系统判断是否达到终止条件。
- 系统在失败时给出下一轮修复路径。

## 6. 关键产品原则

### 6.1 不追求无限发散，优先约束边界

这个系统不应该追求“什么都能聊、什么都能答”。对目标用户来说，更重要的是：

- 能不能把问题限定在可解决范围内。
- 能不能清楚说明当前上下文不足在哪里。
- 能不能避免基于片面信息给出过度自信的答案。
- 能不能把复杂任务拆成可验证步骤。

### 6.2 信息密度要高，但只提供正确的信息

普通用户不需要看到所有代码、日志和底层状态。

产品要做的是提升“有效信息密度”：

- 只展示用户当前需要决策的信息。
- 隐藏不必要的底层噪音。
- 对复杂信息做结构化摘要。
- 把技术输出转化成“目标、步骤、风险、验收、下一步”。

### 6.3 验收链条必须闭合

传统 Chat 模式最大的问题是：LLM 会告诉用户“这样修就好了”，但它未必接入运行环境、生产环境、完整日志和真实上下文。

因此，系统不能只输出修复建议，还必须输出：

- 修复后应该看到什么现象。
- 看到什么现象才说明修好了。
- 如果没有出现预期现象，下一步检查什么。
- 如果出现另一类现象，可能对应哪种替代方案。
- 当前判断的依据和不确定性在哪里。

产品的理想输出不是单一路径答案，而是更可靠的探索结构：

1. 最大概率修复方案。
2. 对应验收标准。
3. 失败时的第二路径。
4. 失败时的第三路径。
5. 每一路径的观察指标和终止条件。

### 6.4 软件工程工作流要产品化

系统需要把成熟的软件工程工作流内化进产品：

- 目标澄清。
- 范围约束。
- 任务拆解。
- todo list。
- TDD / PDD / 测试优先思想（原文缩写待核对）。
- 执行过程跟踪。
- QA 准出。
- 验收标准。
- 迭代终止条件。

这一部分可以吸收 Superpowers 的工作流思想。对大多数目标用户来说，具备完整的软件工程思维工作流，已经足够形成明显价值。

## 7. 产品形态选项

产品形态需要分阶段思考。

第一期不一定要覆盖所有入口，但选型时必须判断底座能否承载未来扩展。

可能形态包括：

- Web 端：长对话、任务管理、可视化流程、验收结果展示。
- IDE 插件：例如 VS Code 插件，用于更专业用户的代码上下文接入。
- CLI：适合开发者或内部团队使用。
- 桌面 GUI：降低安装和本地环境使用门槛。
- IM 接入：飞书、企业微信等，用于企业内部协作和轻量任务入口。
- 工具调用体系：接入代码仓库、运行环境、日志系统、工单系统、CI/CD 等。

关键不是一期做哪个入口，而是：

**我们选择的 harness 底座有没有能力承载这些形态的延展。**

如果底座只能交付一次，后续无法演进，那么产品很快会被限制住。

## 8. 技术选型关注点

### 8.1 LLM 后端

后端 LLM 需要以主力模型为基准，再配合其他国内开源或商业模型进行能力补充。

当前可考虑的方向包括：

- DeepSeek 作为主力模型之一（原文“deep sick”应为 DeepSeek）。
- MiniMax、GLM、Qwen 等国内模型作为补充（具体版本待核对）。
- 自有模型后端服务体系。
- 后续再考虑模型路由、调度、编排和多模型协同。

第一阶段可以暂不把复杂调度作为核心问题，而是先聚焦上层 Agents Harness System 的产品闭环。

### 8.2 Harness 底座

底座需要评估：

- 是否支持任务拆解。
- 是否支持工具调用。
- 是否支持上下文管理。
- 是否支持长任务执行。
- 是否支持执行过程可观测。
- 是否支持测试和验收机制。
- 是否支持多入口扩展。
- 是否支持企业级部署和合规要求。
- 是否便于二次开发和定制化。

OpenHands 可作为起点，但需要和 Codex、Claude Code、Cursor、以及其他闭源方案进行能力对照。

## 9. MVP 建议

第一期不应追求完整平台，而应验证最核心的闭环：

**用户输入一个工程目标，系统能把它转化为可执行、可跟踪、可验收的工程任务。**

建议 MVP 包含：

1. 统一 Web 入口。
2. 目标输入与澄清。
3. 任务拆解。
4. todo list 生成。
5. 测试/验收标准生成。
6. 执行过程状态展示。
7. 结果验收与失败分支建议。
8. 基础工具调用能力。

先不做或弱化：

- 复杂多模型路由。
- 完整 IDE 插件生态。
- 飞书/企业微信入口。
- 高度自动化生产环境接入。
- 过重的 skill 体系复杂度。

## 10. 需要继续确认的问题

### 用户与场景

- 第一批用户到底是内部团队、企业客户，还是外部 SaaS 用户？
- 他们最常见的前三类任务是什么？
- 他们是否愿意提供代码仓库、日志、运行环境等上下文？
- 他们对自动执行的信任边界在哪里？

### 产品形态

- 第一期主入口是 Web、IDE 插件，还是二者并行？
- 用户是否需要“完全看不见代码”的体验？
- 哪些信息必须展示，哪些信息必须隐藏？
- 任务状态、验收标准、失败路径如何可视化？

### 技术底座

- OpenHands 能否满足二次开发、企业部署和多入口扩展？
- 是否需要同时验证 Codex / Claude Code / Cursor 类产品能力作为对照？
- 工具调用体系第一期接哪些工具最有价值？
- 是否要先做一个内部 dogfood 版本？

### 商业交付

- 我们卖的是工具、平台，还是带服务的工程能力？
- 交付边界在哪里？
- 如何定义“完成一次工程任务”的可计费或可衡量指标？
- 合规和数据隔离要求如何影响部署形态？

## 11. 当前阶段结论

这份思考的核心不是“选择哪个开源项目”，而是要确认一个更大的产品判断：

**我们要把 coding agent 能力从专家工具，转化为可被普通/半专业用户使用的工程任务系统。**

产品的关键竞争力不只来自模型能力，而来自：

- 自有算力与模型后端。
- 对目标用户复杂度的控制。
- 对工程工作流的产品化。
- 对测试、验收、准出的闭环设计。
- 对未来多入口、多工具、多模型扩展的承载能力。

下一步最重要的工作，是把这套思考进一步压缩成：

1. 一页产品定位。
2. 三个核心用户场景。
3. 一条 MVP 工作流。
4. 一张底座选型评估表。
