OpenHands Explained

它不是一个神秘黑盒，也不只是一个 benchmark harness。

更准确地说，software-agent-sdk 是 OpenHands 的能力底座，当前主仓是把这些能力做成 GUI、Cloud、Enterprise 的应用层，而 OpenHands/benchmarks 则是独立的评测 harness。第一次接触时，最重要的不是背目录，而是先知道“我进来能做什么”，以及“这个仓库到底管什么”。

一句话心智模型

OpenHands 更像一个带执行环境的 AI 软件工程工作区：你在 Chat 里下任务，它在 sandbox 里做事，再通过 Terminal、Changes、Browser、VS Code 等面板把过程和结果展示给你。

3 层 产品层、能力层、评测层
2 个外部核心仓 SDK 与 Benchmarks 不在主仓内部
1 个主应用仓 当前仓库承载 GUI、app server、enterprise 扩展
60 Seconds Tour

第一次打开 OpenHands，你会看到什么

先别急着看源码。先把它想成一个“会下指令、会执行、会展示过程”的工作区，而不是一个单纯聊天框。

Chat

你告诉它要做什么

在这里输入任务、目标、约束和补充说明。这里不是最终结果页，而是你和 agent 的工作入口。

Changes

你检查它到底改了什么

如果它声称修了 bug、加了功能、更新了页面，你会在这里看到文件差异和代码变更。

Terminal

你确认它有没有真的跑起来

命令执行和输出在这里最直观。它是 OpenHands 不只是“会说”，而是真的“会做”的关键证据。

VS Code

你把工程当项目看

当任务涉及多文件或复杂目录时，这里帮助你像看一个真实代码库那样浏览工作区。

Browser / App

你直接看运行结果

如果任务生成了网页、启动了应用或改了前端，这里能直接看到活的页面，而不是只看文字描述。

Planner

你了解它准备怎么做

在某些模式下，OpenHands 会先展示计划或步骤分解，让你在执行前先判断方向对不对。

Repo Boundary

  • 提供 GUI：首页、会话页、设置页、共享页、组织相关页面。
  • 串联 sandbox、conversation、event、settings、integrations 和 enterprise 扩展。
Conversation Flow

一次会话是怎么跑起来的

理解 OpenHands 最好的方法，不是先背目录，而是沿着一次真实任务从界面走到执行环境，再走回结果展示。

Step 1 你进入界面

前端先知道当前部署模式、有哪些 provider、哪些功能被启用，以及你的基础设置是什么。

Step 2 你创建 conversation

conversation 是“一次任务的主档案”。系统后续所有状态、事件和执行环境都会围绕它组织。

Step 3 app server 串起上下文

它把用户、settings、secrets、conversation、sandbox 入口和事件流通道拼成一次可运行任务。

Step 4 sandbox 真正干活

执行命令、改文件、跑应用、操作仓库，都发生在隔离执行环境里。

Step 5 event 记录过程

系统把运行中的状态、动作和输出组织成事件，让这些过程可以被展示、回放和共享。

Step 6 前端持续同步

WebSocket 和查询层把运行中的变化持续送回 Chat、Terminal、Changes 等面板，而不是最后一次性展示。

Step 7 你检查结果

你在 Chat 看解释，在 Terminal 看执行，在 Changes 看差异，在 Browser / App 看运行结果。

Step 8 结果回到工作流

在 Cloud / Enterprise 场景中，结果还能继续接回 GitHub、GitLab、Slack、Jira、Linear 等系统。

Core Modules

核心模块拆解：把术语翻译成人话

每个模块都用同一套问题来解释：它是什么、你什么时候会遇到它、为什么它关键。

Conversation Layer

app_conversation/

任务主线管理

它是什么：会话管理层。

你什么时候会遇到它：创建任务、打开历史会话、共享会话、恢复会话时。

为什么它关键：它把“用户眼里的一次任务”变成系统里能跟踪、能恢复、能共享的主线。

Execution Layer

sandbox/

隔离执行环境

它是什么：隔离执行环境管理层。

你什么时候会遇到它：agent 需要跑命令、改文件、启动项目、访问运行环境时。

为什么它关键：没有它，OpenHands 只能“说”，很难真正“做”。

Observation Layer

event/

把执行过程变成可展示过程

它是什么：事件存储和流转层。

你什么时候会遇到它：界面里出现状态更新、输出流、会话回放和共享内容时。

为什么它关键：因为 OpenHands 需要把“运行过程”变成“可观察过程”。

Configuration Layer

settings/ + secrets/

配置和凭据管理

它们是什么：模型配置、用户设置、应用设置和敏感信息管理层。

你什么时候会遇到它们：填 API key、配模型、切开关、管理应用行为时。

为什么它们关键：产品不能靠写死配置运行，尤其还要支持 Local GUI、Cloud 和 Enterprise。

Identity Layer

user/ + services/

身份与上下文

它们是什么：用户身份、用户上下文和辅助服务层。

你什么时候会遇到它们：登录、访问控制、provider token 关联、sandbox 访问控制时。

为什么它们关键：系统必须知道“谁在发起请求、谁能访问哪些能力”。

Front-end Boot Layer

web_client/

前端启动配置

它是什么：前端启动配置层。

你什么时候会遇到它：一打开前端，系统就要知道当前是 OSS 还是 SaaS、有哪些 provider、哪些功能启用。

为什么它关键：它决定前端一开始怎么呈现自己。

Workflow Integration

integrations/ + enterprise/

接入真实协作系统

它们是什么：外部系统集成与企业扩展层。

你什么时候会遇到它们：当你希望 agent 与 GitHub、GitLab、Slack、Jira、Linear 或组织级能力结合时。

为什么它们关键：因为 OpenHands 的目标不是孤立运行，而是接回真实团队工作流。

Common Misreads

最容易出现的误解

如果先把这些误解排掉，再去读源码，会省掉大量无效理解成本。

误解 1：这个主仓就是全部 OpenHands

不是。它是最重要的应用层主仓，但不是全部底层能力的唯一载体。

误解 2：OpenHands 就是 benchmark harness

不是。benchmark harness 在外部 `OpenHands/benchmarks` 仓里。

误解 3：OpenHands 只是前端套壳调模型

不是。conversation、sandbox、event、settings、user、integrations 这些层说明它是完整工作区系统。

误解 4：skills 和 microagents 是两套无关系统

不是。它们是 V0 / V1 术语演化中的同一条能力组织脉络。

如果你是第一次接触，推荐路线是：先读这页建立总图，再看 `frontend/README.md` 与 `openhands/app_server/README.md`， 然后再下钻到 `sandbox/`、`app_conversation/`、`event/` 三个模块。想看底层 agent 能力时，再跳到 SDK；想看评测时，再跳到 Benchmarks。
